## 验证步骤

### 步骤1：验证控制台无错误
1. 打开浏览器开发者工具（F12）
2. 切换到Console面板
3. 刷新招聘流程管理页面
4. 确认没有400错误（特别是关于hire_position的错误）

### 步骤2：验证重复检测功能
1. 点击"🔍 检测重复数据"按钮
2. 如果存在重复数据，应弹出报告窗口
3. 测试"合并此组"或"一键合并所有"功能
4. 合并后再次检测，应显示"没有重复记录"

### 步骤3：验证强制同步功能（关键测试）

#### 测试3.1：同步锁机制
1. 打开浏览器开发者工具
2. 快速连续点击2次"强制同步数据"按钮
3. 第二次点击应提示"同步操作正在进行中，请稍后再试"
4. 控制台应显示警告信息

#### 测试3.2：同步不产生重复数据
1. 记录当前记录数（页面右上角显示"共 X 条记录"）
2. 点击"强制同步数据"按钮
3. 等待同步完成
4. 刷新页面
5. 检查记录数是否增加合理（只应增加真正的新记录）
6. 检查是否有重复姓名出现

#### 测试3.3：同步前自动清理
1. 如果存在重复数据，强制同步时应自动清理
2. 控制台应显示："【同步前清理】成功清理 X 条重复记录"

### 步骤4：验证数据准确性
1. 检查统计卡片数字（总应聘人数、初试、复试等）
2. 使用筛选功能（按岗位、环节、状态）
3. 点击"导出Excel"，检查导出数据是否准确

## 预期结果

### ✅ 正常情况
- 同步完成后，记录数合理增加
- 无重复数据产生
- 控制台无错误
- 统计数字准确

### ⚠️ 需要注意的情况
- 如果历史重复数据较多，首次同步可能需要较长时间
- 数据库唯一约束需要在Supabase控制台手动执行SQL

## 问题排查

### 如果仍然产生重复数据
1. 检查控制台输出：
   - 是否显示"按姓名+手机号/身份证号查找的记录数: X"（X应大于0）
   - 是否显示重复检测相关的日志

2. 检查数据库约束是否已添加：
   - 登录Supabase控制台
   - 执行：\`SELECT indexname FROM pg_indexes WHERE tablename = 'recruitment_process';\`
   - 确认存在 `idx_unique_name_phone` 等索引

### 如果同步锁不生效
1. 检查控制台是否显示"同步操作正在进行中"
2. 快速连续点击按钮测试

---

## 数据库约束执行说明

### 执行SQL文件
1. 登录Supabase控制台
2. 进入SQL编辑器
3. 打开文件：`sql/add_unique_constraints.sql`
4. 执行SQL语句
5. 如果报错（存在重复数据），先使用"检测重复数据"功能清理

### 验证约束
```sql
-- 查看已创建的索引
SELECT indexname, indexdef 
FROM pg_indexes 
WHERE tablename = 'recruitment_process' 
AND indexname LIKE 'idx_unique_%';
```
